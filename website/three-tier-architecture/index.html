This is a Three-Tier Architecture, containing a Presentation tier, an Application tier and a Data tier. The presentation tier contains the user-facing elements. The application tier contains the back end, which would also contain the application code (not present here). And the data tier, containg the application's data, normally in the form of a database.

A three-tier architecture makes the system more organised and more easily maintainable. Each tier can be developed and scaled independently based on requirements. This architecture also provides increased security, the database can be isolated behind additional layers of security provided by the front end.

This architecture has high availability and fault tolerance built into it as it is spread across different availability zones, and it uses a load balancer and auto scaling. This means disruption will be kept to a minimum by the automatic provisioning of resources to ensure the application always has the right amount of capacity to handle current demand.

Services Used

EC2, EC2 Auto Scaling, Elastic Load Balancing, RDS

  1. VPC: Created across 2 Availability Zones, with public and private subnets for each tier.
  2. Public Subnets: Auto-assigned public IPv4 addresses for internet access via a public route table and internet gateway.
  3. NAT Gateways: Created in each Availability Zone for private subnet instances to access the internet (e.g. for updates and patches). Assigned an Elastic IP to each NAT Gateway.
  4. Route Tables: Configured for each subnet. Public subnets with a route to an internet gateway, and private subnets with a route to the NAT Gateway.
  5. Presentation Tier: Launched on EC2 instances (configured using a launch template) via an Auto Scaling Group with a Load Balancer distributing traffic. Security group allows SSH, HTTP, HTTPS. The instance user data contains a script which installs a web server, displaying some basic HTML.

    The Auto Scaling Group can dynamically provision EC2 instances when needed across multiple availability zones in the public subnets. This ensures high availability and limits single points of failure. Here, I created a dynamic scaling policy which monitored CPU usage.
  6. Application Tier: Similar setup to presentation tier, but with a bastion host for private internet access, and a connection to the database. Security group restricts public access but allows SSH/ICMP from presentation tier. Launched via Auto Scaling Group with an internal Load Balancer in private subnets.
  7. In the launch template instance user data, I entered a script to install the MySQL package, to allow for later connection to the database in the data tier.
  8. Bastion Host: Created using an EC2 instance, in a public subnet, to allow for private connection to the internet, by authorized users, via SSH, using SSH Agent Forwarding.
  9. Data Tier: Using Amazon RDS for MySQL with a database security group allowing connection from the application tier security group. I configured the inbound and outbound traffic to run to and from the servers running in the application tier, over port 3306.

    Here, I created the RDS database in a single availability zone. For best practice you would enable Multi-AZ deployment, providing the database with high availability and automatic failover depending on configuration. You can then also create read replicas for more efficient read queries.
